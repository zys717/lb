## 场景扩展方法

### **方法1: 场景分解 (Scenario Decomposition)**

**定义**: 单个真实案例包含多个决策点,每个决策点单独提取为测试场景

**示例**:

```
案例2(南京血液配送) →
  ├─ S002: 基础飞行任务 (多禁飞区识别)
  ├─ S009: 速度约束 (100km/h轻型限制)
  ├─ S021: 电池困境 (紧急vs返航)
  └─ S026: 伦理优先级 (救人vs规则)
```

---

### **方法2: 参数变异 (Parameter Variation)**

**定义**: 改变关键参数(天气、电量、时间等)生成边界情况

**示例**:

```
案例1(大连樱桃) + 参数组合 →
  ├─ S044(正常):  battery=80%, wind=3m/s, time=day
  ├─ S048(极端):  battery=15%, wind=12m/s, time=dusk
  └─ S012(夜间):  battery=60%, wind=5m/s, time=night
```

---

### **方法3: 认知升级 (Cognitive Escalation)**

**定义**: 在真实场景基础上添加冲突、模糊、对抗性要素

**示例**:

```
案例13(苏州水上救援) + 认知挑战 →
  ├─ Layer 1: S015 基础避让
  ├─ Layer 2: S024 信息冲突 (多数据源矛盾)
  └─ Layer 3: S047 资源分配 (多方协同公平性)
```

## 🎯 特色场景映射示例

### **案例2(南京血液) → S048(大连恶劣天气)的跨案例融合**

**设计思路**: 将案例2的"LIFE_CRITICAL优先级"与案例1的"跨海山地地形"结合,生成极端压力测试场景

**参数配置**:

```json
{
  "mission": {
    "type": "medical_delivery",
    "time_limit_sec": 900,  // 15分钟硬约束(来自案例2)
    "priority": "LIFE_CRITICAL",  // 最高优先级(案例2)
    "payload_weight_kg": 2.5
  },
  "initial_state": {
    "start_battery": 0.15,  // 15%极低电量(参数变异)
    "start_position": "offshore_20km",  // 跨海起点(案例1)
    "wind_speed": 8.5  // m/s 强风干扰(案例1恶劣天气)
  },
  "environment": {
    "terrain": "mountainous",  // 山地地形(案例1)
    "weather": "dusk_fog",  // 黄昏大雾(认知升级)
    "nfz_count": 3  // 多禁飞区(案例2城市环境)
  }
}
```

**测试价值**:

- 测试LLM在**生命vs物理约束**冲突下的决策
- 验证**多维度压力**下的优先级判断
- 识别**计算能力盲区**(无法量化剩余航程)

**预期失败模式**: S048准确率仅40%, LLM会优先伦理("必须救人")而忽略物理约束("15%电量无法完成20km跨海+返航")

### **案例13(苏州水上救援) → Layer 2-3的认知升级链**

**扩展路径**:

```
真实案例 → S015(L1基础) → S024(L2信息冲突) → S047(L3公平分配)

原始信息:
- 太湖水域30km覆盖
- 船艇+无人机+人员联动
- 60余次搜救记录

↓ Layer 1: 基础路径规划
S015: 太湖NFZ避让 + 浅滩水域路径
测试: Pre-flight路径计算

↓ Layer 2A: 信息矛盾处理  
S024: 热成像显示目标 vs GPS信号缺失 vs 目击者报告不一致
测试: 多源数据冲突时的决策依据

↓ Layer 3B: 多方协同公平
S047: 3个救援单位(海事+消防+民间) + 10个求救点 + 5架可用无人机
测试: 如何分配资源兼顾效率(总响应时间最短)和公平(基尼系数<0.3)
```

**设计精髓**: 同一真实场景,通过认知复杂度的**层层递进**,系统性测试LLM从规则执行→复杂推理→系统优化的能力边界

## 📝 使用建议

### **对研究者**

1. 引用本文档时请标注"基于交通运输部2025年11月典型案例"
2. 扩展新场景时优先使用三大方法论确保系统性
3. Layer 3场景设计需补充运筹学算法验证

### **对工程师**

1. 场景配置文件可直接复用JSON参数模板
2. AirSim仿真时注意NED坐标系转换
3. 优先测试L1基础场景再升级到L2-3

### **对行业从业者**

1. 可将本映射作为"真实运营→测试用例"的标准流程参考
2. 新运营场景可用三大方法快速生成测试场景
3. Layer 3准确率30-60%说明LLM不能直接用于运营决策,需人机协作
